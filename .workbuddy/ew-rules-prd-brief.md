## 一、项目背景

### 现有网站
- **URL**: https://kokkotham.github.io/ash-world
- **技术栈**: 静态 HTML + Three.js（3D 粒子背景）+ Ontology 数据系统
- **Ontology 体系**:
  - `data/races.json` — 种族数据（ancient + modern 两类）
  - `data/deities.json` — 神祇数据
  - `data/modules.json` — 模组数据
  - `data/regions.json` — 地区数据
  - `data/links.json` — 跨实体关联数据

- **现有页面**: races / pantheon / worldview / map / rules / character-sheet / glossary / modules / news / articles / community / art / discussion / merch / works

## 三、核心目标

1. **docx → MD → JSON 转换流水线**：将 65 个 docx 转为结构化 JSON，融入现有 Ontology 数据体系
2. **规则书主页面**：增强现有 `rules.html`，作为所有规则内容的导航入口
3. **章节详情页**：种族/专修/神术/故事运作 四个维度的独立页面
4. **关键词悬浮提示**：鼠标悬停关键术语时显示释义 tooltip（复用/扩展 glossary 体系）
5. **Git 追踪**：所有内容变更通过 Git 追踪

## 四、PRD 输出要求

请按 **简单 PRD 模板** 输出，包含以下章节：

### 4.1 产品目标
- 这个规则书 Web 系统的核心价值是什么
- 解决现有 docx 文件的什么问题

### 4.2 用户故事（3-5 个）
- 典型用户场景，如：
  - "作为玩家，我想浏览可选种族找到想玩的角色"
  - "作为主持人，我想快速查阅专修规则"
  - "作为新人，我想通过悬浮提示理解术语"

### 4.3 需求池（P0/P1/P2 分级）
- P0 = 必须实现
- P1 = 应该实现
- P2 = 锦上添花

### 4.4 UI 设计稿
- 规则书主页结构描述
- 章节详情页布局
- 关键词悬浮提示交互
- 与现有网站导航的衔接方式

### 4.5 与现有 Ontology 体系的融合方案
- 如何扩展 `races.json`（新增 EW 种族字段）
- 是否需要新建 `specialties.json` / `spells.json` / `combat-rules.json`
- 如何扩展 `links.json` 关联规则实体
- 术语表 `glossary.html` 如何扩展

### 4.6 待确认问题
- 需要澄清的设计/技术决策
- 与现有内容的冲突/覆盖策略

### 4.7 数据转换策略
- docx → Markdown → JSON 的转换方案
- 批量转换 vs 逐个转换
- 质量保证机制
