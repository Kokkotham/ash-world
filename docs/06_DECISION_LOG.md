> **最后更新**: 2026-07-04

## ADR-001: 为什么使用 Vue 3？

**决策**: 前端框架选择 Vue 3。

**理由**:
- 组件化开发，维护方便
- 学习曲线平缓，适合小团队
- 生态成熟，Pinia / Vue Router / Vite 配套完整
- 单文件组件 (SFC) 结构清晰，模板/逻辑/样式分离
- 响应式系统天然适合数据驱动的规则展示

**否决方案**:
- React: 学习成本高，JSX 对非前端开发者不友好
- Svelte: 生态不够成熟，组件库少
- 纯静态 HTML (当前): 无法支撑复杂交互（角色卡计算、搜索等）

## ADR-002: 为什么使用 Django + DRF？

**决策**: 后端选择 Django + Django REST Framework。

**理由**:
- Django Admin 成熟，自带后台管理界面
- SimpleUI 插件美化 Admin，开发组可以直接在后台编辑规则
- DRF 序列化器、视图集、权限系统完善
- ORM 强大，适合规则之间的复杂关联查询
- Python 生态丰富，方便编写数据迁移脚本

**否决方案**:
- Node.js (Express/Koa): 无内置 Admin，需额外开发后台
- Flask: 太轻量，Admin 需从零搭建
- CloudBase Server (当前): 无关系型查询，关联查询困难

## ADR-003: 为什么使用 MySQL？

**决策**: 数据库选择 MySQL 8.0+。

**理由**:
- 规则之间关联复杂（规则↔词条↔世界观↔角色卡）
- 关系型数据库更适合多对多关联查询
- 全文索引支持规则搜索
- 事务保证数据一致性（角色卡修改 + 日志记录）
- 运维生态成熟，云数据库成本低

**否决方案**:
- CloudBase NoSQL (当前): 关联查询困难，无事务
- PostgreSQL: 功能更强但运维成本略高
- SQLite: 不适合生产环境多用户并发

## ADR-004: 为什么不用 Markdown 规则书？

**决策**: 不使用 Markdown 文件存储规则，使用数据库。

**理由**:
- 以后角色卡需要引用规则（如"你的种族提供 +1 力量"），数据库才能自动计算
- 规则之间有关联关系（如"专修 A 的 B 技能参见词条 C"），数据库可以做关联查询
- 版本管理需要 diff 对比，数据库可以存快照
- 后台编辑需要富文本编辑器，SimpleUI + Django CKEditor 可实现
- 搜索功能需要全文索引，数据库天然支持

**否决方案**:
- Markdown 文件: 无法关联查询、无法自动计算、无版本快照
- 纯 JSON 文件 (当前原型): 可以渲染但无查询能力，无事务，无权限控制

## ADR-005: 为什么保持黑金暗黑幻想风格？

**决策**: 全站统一黑金暗黑幻想视觉风格，禁止改变。

**理由**:
- 这是 Embers World 的品牌视觉识别
- TRPG 规则书的氛围需要沉重、神秘、古典的感觉
- 黑金色与"灰烬"、"余烬"的主题词高度契合
- 统一风格降低用户认知成本

**否决方案**:
- 蓝色科技风: 与 TRPG 桌游调性冲突
- 玻璃拟态: 过于现代，破坏沉浸感
- Material Design: 谷歌风格，不搭
- 苹果风: 过于简洁轻快

## ADR-006: 为什么前后端完全分离？

**决策**: Vue + Django REST API，禁止前端直接操作数据库。

**理由**:
- 前端关注 UI，后端关注业务逻辑，职责清晰
- API 可复用（未来移动端、开放 API 都能用）
- 安全性更高（数据库不暴露给前端）
- 可独立部署和扩展

**否决方案**:
- Django 模板渲染 (SSR): 交互体验差，不适合 SPA
- Vue 直连数据库 (当前 CloudBase 模式): 安全性差，业务逻辑散落在前端

## ADR-007: 为什么使用 SimpleUI？

**决策**: Django Admin 使用 SimpleUI 美化。

**理由**:
- 开发组（非程序员）可以直接在后台编辑规则
- SimpleUI 提供现代化 UI，比原生 Django Admin 友好
- 零开发成本，安装即用
- 支持富文本编辑器、图片上传等插件

**否决方案**:
- 自研后台: 开发成本高，Phase 1 不值得
- xadmin: 维护停滞，不兼容 Django 5.0
- 原生 Admin: UI 过时，开发组使用门槛高

## ADR-008: 为什么 CloudBase 作为 Phase 0？

**决策**: 原型阶段使用 CloudBase (静态托管 + Auth + NoSQL)。

**理由**:
- 零后端运维，快速验证产品
- 自带手机号验证码登录，不需要配置 SMS
- 静态托管 + CDN，全球加速
- 与腾讯云生态打通（域名、DNS）

**迁移原因 (Phase 1)**:
- NoSQL 关联查询困难
- 无事务支持
- 无自定义后台管理
- 无全文搜索
- 厂商锁定风险

## ADR-009: 为什么不做浅色主题？

**决策**: 全站只有暗色主题，不做 light/dark 切换。

**理由**:
- 暗黑幻想风格只在暗色下成立
- 浅色版会破坏品牌一致性
- 节省开发成本（不需要维护两套色板）
- TRPG 玩家长时间阅读规则，暗色更护眼

---

## ADR-010: 为什么用 Docker 部署？

**决策**: Phase 1 使用 Docker + docker-compose 部署。

**理由**:
- 环境一致性（开发/测试/生产）
- 一键启动 Vue + Django + MySQL + Nginx
- 方便回滚和扩容
- 与 CI/CD 集成方便

**否决方案**:
- 手动部署: 环境不一致，维护成本高
- 云函数 (CloudBase Functions): 冷启动慢，不适合 Django
- 裸机部署: 配置繁琐，不可复现
